# Systematic Debugging - 系统性调试

## 概述

随机修复浪费时间并产生新 bug。快速补丁只会掩盖底层问题。

**核心原则:** 在尝试修复之前，必须先找到根因。治标不治本就是失败。

**违反这个流程的字面意思就是违反调试的精神。**

## 铁律

```
不经过根因调查，不能尝试任何修复
```

如果你没有完成阶段一，就不能提出修复方案。

## 何时使用

用于任何技术问题：
- 测试失败
- 生产环境 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别是在以下情况：**
- 时间压力大（紧急情况让猜测变得诱人）
- "快速修一下"看起来很明显
- 已经尝试了多次修复
- 之前的修复没有生效
- 你不完全理解问题

## 四个阶段

必须完成每个阶段才能进入下一个。

### 阶段 1: 根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们通常包含确切的解决方案
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误代码

2. **稳定复现**
   - 能可靠地触发吗？
   - 确切的步骤是什么？
   - 每次都发生吗？
   - 如果不能复现 → 收集更多数据，不要猜测

3. **检查最近的变更**
   - 什么变化可能导致这个问题？
   - Git diff，最近的提交
   - 新依赖，配置变更
   - 环境差异

4. **在多组件系统中收集证据**
   ```
   对于每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置传递
     - 检查每一层的状态

   运行一次收集证据，显示在哪里出错
   然后分析证据，找出失败的组件
   然后调查那个特定组件
   ```

5. **追踪数据流**
   - 坏数据从哪里来？
   - 是什么用坏数据调用了这里？
   - 持续向上追踪直到找到源头
   - 在源头修复，而不是在症状处

### 阶段 2: 模式分析

1. **找到可工作的例子**
   - 在同一代码库中找类似的工作代码
   - 什么类似的东西是正常工作的？

2. **与参考对比**
   - 如果实现某个模式，完整阅读参考实现
   - 不要略读 - 读每一行

3. **识别差异**
   - 工作的和坏的有什么不同？
   - 列出每一个差异，无论多小

### 阶段 3: 假设和测试

1. **形成单一假设**
   - 清楚陈述："我认为 X 是根因，因为 Y"
   - 写下来
   - 要具体，不要模糊

2. **最小化测试**
   - 做最小的可能变更来测试假设
   - 一次一个变量
   - 不要同时修复多个问题

3. **验证后继续**
   - 有效？ → 阶段 4
   - 无效？ → 形成新假设
   - 不要在上面堆更多修复

### 阶段 4: 实现

1. **创建失败的测试用例**
   - 最简单的复现
   - 如果可能，自动化测试
   - 修复前必须有

2. **实现单个修复**
   - 解决识别出的根因
   - 一次一个变更
   - 没有"顺便"改进

3. **验证修复**
   - 测试现在通过了？
   - 没有其他测试失败？
   - 问题真的解决了？

4. **如果修复无效**
   - 停止
   - 计数：你尝试了多少次修复？
   - 如果 < 3：返回阶段 1，用新信息重新分析
   - **如果 ≥ 3：停止并质疑架构**

## 红线 - 停下并遵循流程

如果你发现自己在想：
- "先快速修一下，以后再调查"
- "试着改一下 X 看看能不能工作"
- "添加多个更改，运行测试"
- "跳过测试，我手动验证"
- "可能是 X，让我修一下"
- "我不完全理解但这可能有效"

**所有这些都意味着：停止。返回阶段 1。**

## 常见借口

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要流程" | 简单问题也有根因。流程对简单 bug 很快。 |
| "紧急情况，没时间走流程" | 系统性调试比乱试快得多。 |
| "先试这个，然后再调查" | 第一个修复设定模式。从一开始就做对。 |
| "一次多改几个省时间" | 无法隔离哪个有效。会导致新 bug。 |

## 嵌入式领域扩展

**硬件相关调试额外步骤：**
- 使用示波器/逻辑分析仪收集硬件层证据
- 检查电源、时钟、复位信号
- 验证硬件初始化顺序
- 考虑时序和竞态条件

## 真实影响

从调试会话：
- 系统性方法：15-30 分钟解决
- 随机修复方法：2-3 小时瞎折腾
- 首次修复成功率：95% vs 40%
- 引入新 bug：几乎没有 vs 常见
