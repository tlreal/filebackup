# Feasibility Assessment - 可行性评估

## 概述

系统性评估嵌入式项目的技术可行性，识别风险并提出应对策略。

## 评估维度

### 1. 技术可行性

| 维度 | 评估内容 |
|------|---------|
| **硬件能力** | 处理能力、存储、外设是否满足 |
| **软件复杂度** | 算法难度、实时性要求 |
| **团队能力** | 技术储备、经验匹配度 |
| **工具链成熟度** | 开发环境、调试工具 |

### 2. 资源可行性

| 维度 | 评估内容 |
|------|---------|
| **时间** | 开发周期是否合理 |
| **人力** | 团队规模是否足够 |
| **成本** | 预算是否覆盖 BOM + 开发 |
| **供应链** | 芯片/物料是否可获取 |

### 3. 风险识别

**技术风险:**
- 性能是否有余量
- 是否依赖未验证技术
- 硬件兼容性风险

**项目风险:**
- 需求变更可能性
- 依赖第三方进度
- 认证/合规风险

## 评估流程

```
需求分析结果
     ↓
技术可行性评估 → 不可行 → 调整需求/方案
     ↓ 可行
资源可行性评估 → 不可行 → 调整资源/范围
     ↓ 可行
风险评估 → 高风险 → 制定应对策略
     ↓ 可控
评估通过，进入 Plan 阶段
```

## 输出格式

```yaml
项目名称: xxx
评估日期: YYYY-MM-DD
评估结论: 可行 / 有条件可行 / 不可行

技术可行性:
  硬件能力: ✅ 满足 / ⚠️ 临界 / ❌ 不足
  软件复杂度: ✅ 可控 / ⚠️ 有挑战 / ❌ 过高
  团队能力: ✅ 匹配 / ⚠️ 需学习 / ❌ 不足
  
资源可行性:
  时间: ✅ 充足 / ⚠️ 紧张 / ❌ 不足
  成本: ✅ 充足 / ⚠️ 紧张 / ❌ 不足
  供应链: ✅ 稳定 / ⚠️ 有风险 / ❌ 困难

风险清单:
  高风险:
    - 风险1: [描述] → 应对: [策略]
  中风险:
    - 风险2: [描述] → 应对: [策略]

前置条件:
  - [需要满足的前提条件]

建议:
  - [改进建议]
```

## 与 Core Workflow 集成

**在 Think 阶段调用:**
- 在 `requirements-analysis` 完成后
- 作为进入 Plan 阶段的门禁

**输出流向:**
- 评估结果用于 `platform-selection` 决策
- 风险清单用于 `writing-plans` 规划
