# ESP-Thinking - 极简法则·头脑风暴技能

## 概述

在产品定义阶段，用**互动式头脑风暴**方法，与用户一起明确需求范围，生成并筛选出"把复杂留给自己，把简单留给用户"的创意。

**核心原则：** 
- 互动式引导，而非 AI 假设
- 先明确需求边界，再生成创意
- 用极简法则过滤，确保每个创意都能实现成本腰斩或体验倍增
- 输出结构化需求文档

## 何时使用

```
需要定义新产品/功能？
    ↓
应用极简法则头脑风暴
    ↓
输出经过极简过滤的创意池
```

**使用场景：**
- 新产品定义初期
- 现有产品需要简化重构
- 面临成本压力或用户体验问题
- 需要突破性功能创新

**何时不使用：**
- 已有明确的产品方向和需求
- 紧急交付场景（时间不足以支撑完整 brainstorming）
- 单一功能的小幅优化

## 准备阶段

### 1.1 明确议题

**关键动作：** 把问题锁定在"如何用极简方式满足某一核心痛点"

**输出：** 一份清晰的议题陈述

**示例：**
```
❌ 模糊议题："改进我们的产品"
✅ 清晰议题："如何用 50% 的功能满足用户 80% 的核心需求？"
✅ 清晰议题："如何将用户使用步骤从 5 步减少到 2 步？"
```

### 1.2 互动式多视角引导

**核心方法：** AI 作为引导者，用户作为领域专家。逐个视角进行互动提问，而非 AI 假设。

**引导流程：**

```
明确议题 → 业务视角提问 → 用户回答 → 设计视角提问 → 用户回答 → 
技术视角提问 → 用户回答 → 市场视角提问 → 用户回答 → 需求总结确认
```

**各视角关键问题清单：**

| 视角 | 关键问题（每次选择 3-5 个） |
|------|--------------------------|
| **业务视角** | 1. 目标用户是谁？<br>2. 典型使用场景？<br>3. 当前解决方案的痛点？<br>4. 商业模式？<br>5. 目标售价？ |
| **设计视角** | 1. 产品形态偏好？<br>2. 重量/尺寸要求？<br>3. 交互方式（屏幕/语音/按键）？<br>4. 外观风格？<br>5. 防护等级？ |
| **技术视角** | 1. 通信技术选择？<br>2. 连接方式？<br>3. 续航要求？<br>4. 智能功能部署方式（本地/云端）？<br>5. 性能指标？ |
| **市场视角** | 1. 目标售价？<br>2. 核心竞争优势？<br>3. 主要竞品？<br>4. 简化路径偏好（价格/命题）？<br>5. 一句话价值主张？ |

**注意事项：**
- 每个视角提问后，**必须等待用户回答**才能进入下一个视角
- 根据用户回答，可以追问澄清问题
- 记录用户的所有回答，用于后续创意生成

### 1.3 设立"极简钥匙"提示板

预先写下两条极简路径提示：

```
┌─────────────────────────────────────┐
│     极简钥匙提示板                   │
├─────────────────────────────────────┤
│  ① 删功能、降成本（价格简化）        │
│  ② 删步骤、提体验（命题简化）        │
├─────────────────────────────────────┤
│  约束规则：                          │
│  • 只允许提出能让成本腰斩或体验      │
│    倍增的想法                        │
│  • 如果想法同时涉及"低价 + 易用"，   │
│    请分开成两个创意                  │
│  • 任何超出 3 个字的形容词都要改写   │
│    成可量化指标                      │
└─────────────────────────────────────┘
```

## 创意生成

### 核心约束

| 约束 | 做法 | 目的 |
|------|------|------|
| **禁止批评** | 任何人不得点评、质疑、夸奖他人的想法 | 保持想法数量最大化 |
| **鼓励"野蛮"想法** | 先追求数量（目标 ≥ 30 个），再追求质量 | 极简法则强调"把复杂留给自己" |
| **搭便车** | 允许在他人想法上标注"+10"或"×5"改进点 | 促成组合与放大效应 |

### 记录规则

- 每条创意写在卡片上，贴在白板/墙面
- 主持人实时拍照存档，避免讨论过程中删除卡片
- 使用统一格式：`[视角] 创意描述 - 预期影响`

**示例：**
```
[设计] 移除所有二级菜单，改为扁平化导航 - 减少 60% 点击次数
[技术] 使用缓存预加载替代实时查询 - 响应速度提升 3 倍
[业务] 只保留 3 个核心功能，其他全部移除 - 开发成本降低 50%
```

## 整理与极简测试

### 3.1 需求总结与确认

**关键动作：**
- 总结所有视角的用户回答
- 形成需求洞察文档
- **与用户确认需求是否准确**

**输出格式：**
```markdown
## 需求洞察记录

### 业务视角
- 目标用户：[用户回答]
- 使用场景：[用户回答]
- 核心痛点：[用户回答]

### 设计视角
- 产品形态：[用户回答]
- 交互方式：[用户回答]

### 技术视角
- 通信技术：[用户回答]
- 智能功能：[用户回答]

### 市场视角
- 目标售价：[用户回答]
- 核心优势：[用户回答]
```

**❓ 确认问题：** "以上总结的需求是否准确？有没有需要调整或补充的？"

### 3.2 创意生成

**基于确认的需求，AI 生成创意：**

| 视角 | 创意 | 预期影响 | 简化路径 |
|------|------|----------|---------|
| [视角] | [创意描述] | [量化影响] | [价格/命题] |

**生成数量：** 10-15 个高质量创意（基于用户需求，而非空想）

### 3.3 智能功能价值评估（如适用）

**如果产品包含智能功能（如 AI 助手），使用以下框架评估：**

| 评估维度 | 问题 | 决策规则 |
|---------|------|---------|
| **用户价值** | 是否解决实际问题？ | 是→保留，否→删除 |
| **技术可行** | 端云协同是否可行？ | 是→保留，否→简化 |
| **成本可控** | 是否在目标成本内？ | 是→保留，否→简化 |
| **差异化** | 是否形成竞争优势？ | 是→保留，否→删除 |

**区分"真正有价值的智能"vs"为了智能而智能"：**
- ✅ 真正有价值：解决实际问题、用户愿意付费、技术可行
- ❌ 噱头功能：增加成本但价值有限、用户感知弱、技术炫耀

### 3.4 极简测试

**对每个创意进行四问测试：**

```
1. 能否让成本再降 50%？（价格简化）
   → 如果不能，这个创意不够极致

2. 能否让使用步骤再少 50%？（命题简化）
   → 如果不能，这个创意不够用户友好

3. 能否用一个词描述它的核心价值？
   → 如果不能，这个创意不够聚焦

4. 技术/商业是否可行？
   → 如果不能，这个创意需要重新设计
```

### 3.5 倾向测试 & 市场间隙测试

**倾向测试：**
- 判断团队更擅长价格简化还是命题简化
- 评估企业资源、团队专长、品牌基因

**市场间隙测试：**
- 检查市场是否已有同类简化者
- 寻找未被覆盖的简化路径

**输出：** 为后续选择层决策提供数据支持

## 输出与下一步

### 核心产出

**1. 需求洞察文档：**
- 目标用户与场景
- 核心痛点分析
- 产品形态定义
- 技术方案要点
- 市场定位
- 智能功能评估（如适用）

**2. 极简创意清单：**
- 通过测试后保留的 5-7 个"极简钥匙"创意
- 每个创意包含：问题、解决方案、预期影响、简化路径

**3. 功能裁剪/用户旅程草稿：**
- 为设计层准备的初稿
- 确保"复杂留给自己"

**4. 需求文档输出：**
- 将以上所有内容写入 `docs/plans/YYYY-MM-DD-<topic>-requirements.md`
- 文档包含完整的需求洞察、创意清单、技术方案、市场定位

**5. 行动清单：**
- 启动最小可行原型（MVP）
- 安排快速用户测试
- 开始专属系统设计

### 自动流程：询问下一步

**完成头脑风暴后，必须执行以下询问流程：**

```
输出所有创意和测试结果
    ↓
询问用户：
    ↓
"头脑风暴已完成！我们生成了 [X] 个创意，经过极简测试保留了 [Y] 个核心创意。

接下来是否要执行 ESP-Plan（规划技能），确定产品的简化路径并设计产品架构？

回复'是'或'开始'将继续执行 Plan 技能。"
    ↓
等待用户确认
    ↓
如果用户确认 → 自动调用 ESP-Plan 技能
如果用户拒绝 → 暂停，等待进一步指示
```

**询问话术示例：**
```
✅ 头脑风暴已完成！

本次成果：
- 共生成创意：32 个
- 通过极简测试：6 个核心创意
- 简化路径建议：命题简化（基于团队 UX 能力强）

接下来是否要执行 ESP-Plan（规划技能），确定产品的简化路径并设计产品架构？

回复"是"或"开始"将继续执行 Plan 技能。
```

**注意事项：**
- 必须等待用户明确确认后才能继续
- 如果用户拒绝，记录原因并暂停
- 如果用户要求修改，先满足修改需求再询问

## 常见错误

1. **24 小时复盘：** 每次 brainstorming 后 24 小时内复盘，记录哪些约束被打破，下一次如何调整

2. **重新定义议题：** 若测试显示两条极简路径均不适用，立即重新定义议题，再次 brainstorm

3. **换视角激发：** 当创意池枯竭时，用"六何法"（何人、何时、何地、何事、为何、如何）重新激发灵感

## 常见错误

| 错误 | 表现 | 修复 |
|------|------|------|
| **过早评判** | 在创意生成阶段就开始点评 | 设立"禁止批评"规则，指定主持人监督 |
| **数量不足** | 只生成 10-15 个创意就停止 | 设定硬性目标≥30 个，用计时器逼迫 |
| **混合路径** | 一个创意同时追求低价和易用 | 强制拆分成两个独立创意 |
| **模糊描述** | 使用"更好"、"更快"等模糊词汇 | 要求所有形容词必须可量化 |
| **缺乏过滤** | 没有应用极简三问就进入下一步 | 严格执行三问测试，不通过的创意归档但不进入下一轮 |

**案例 2：电商平台重构**

- Brainstorming 后：2 步完成购买

**案例 3：硬件产品成本优化**
