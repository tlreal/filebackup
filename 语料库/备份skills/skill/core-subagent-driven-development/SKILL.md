# Subagent-Driven Development - 子代理驱动开发

## 概述

通过为每个任务派发新鲜的子代理来执行计划，每个任务完成后进行两阶段审查：先规格符合性，后代码质量。

**核心原则:** 新鲜子代理/任务 + 两阶段审查（规格+质量） = 高质量，快速迭代

## 何时使用

- 有实施计划
- 任务大多独立
- 希望在当前会话中完成

**vs. Executing Plans（并行会话):**
- 同一会话（无上下文切换）
- 每个任务新鲜上下文（无上下文污染）
- 每个任务后两阶段审查
- 更快迭代（任务间无需人工干预）

## 流程

```
读取计划 → 提取所有任务
    │
    ▼
┌─────────────────────────────────────────────────┐
│  Per Task Loop:                                  │
│                                                  │
│  1. 派发实现子代理                               │
│     └→ 子代理有问题？回答后继续                  │
│     └→ 子代理实现、测试、提交、自审              │
│                                                  │
│  2. 派发规格审查子代理                           │
│     └→ 不符合规格？实现者修复 → 再审             │
│                                                  │
│  3. 派发代码质量审查子代理                       │
│     └→ 质量问题？实现者修复 → 再审               │
│                                                  │
│  4. 标记任务完成                                 │
└─────────────────────────────────────────────────┘
    │
    ▼
所有任务完成 → 最终代码审查 → finishing-branch
```

## 两阶段审查

### 阶段 1: 规格符合性审查

确认实现**完全匹配**规格：
- 没有遗漏功能
- 没有额外功能（YAGNI）
- 接口符合设计

### 阶段 2: 代码质量审查

确认代码质量：
- 可读性
- 可维护性
- 测试覆盖
- 无明显问题

**重要:** 必须先通过规格审查，再进行质量审查。

## 红线

**绝不:**
- 跳过任何审查（规格或质量）
- 带着未修复的问题继续
- 并行派发多个实现子代理（会冲突）
- 让子代理读取计划文件（直接提供完整任务文本）
- 在规格符合性通过前开始质量审查
- 在任一审查有未解决问题时进入下一任务

**如果子代理问问题:**
- 清楚完整地回答
- 提供必要的额外上下文
- 不要催促他们进入实现

**如果审查发现问题:**
- 实现者（同一子代理）修复
- 审查者再次审查
- 重复直到通过
- 不要跳过再审

## 领域扩展钩子

**如果是嵌入式项目:**
- 实现子代理应参考 `embedded-driver-development` 等领域 skills
- 审查应考虑嵌入式特有要求（内存、功耗、实时性）

## 集成

**必需的工作流 skills:**
- `core-writing-plans` - 创建此 skill 执行的计划
- `core-code-review` - 审查子代理使用的审查模板
- `core-finishing-branch` - 所有任务完成后收尾

**子代理应使用:**
- `core-test-driven-development` - 每个任务遵循 TDD
