# ESP-Document - 极简法则·产品定义文档技能

## 概述

在完成头脑风暴、规划和执行规划后，将整个过程输出为一份完整的产品定义文档。这份文档是产品开发的指导性文件，确保团队对产品的理解一致。

**核心原则：** 文档必须简洁、可执行、可验证，体现极简法则的核心思想。

## 何时使用

```
完成 ESP-Thinking?
    ↓
完成 ESP-Plan?
    ↓
完成 ESP-Act?
    ↓
运行 ESP-Document
    ↓
输出产品定义文档
```

**使用场景：**
- 完成 ESP 三阶段后需要正式文档
- 需要向团队/管理层汇报产品定义
- 需要为开发团队提供清晰指引
- 需要存档和后续回溯

**何时不使用：**
- ESP 三阶段尚未完成
- 仅需临时沟通而非正式文档
- 文档受众已充分了解产品定义

## 文档结构

### 标准模板

```markdown
# [产品名称] 产品定义文档

## 1. 执行摘要 (Executive Summary)
- 一句话产品定义
- 简化路径宣言
- 核心价值主张

## 2. 市场洞察 (Market Insights)
- 目标用户画像
- 核心痛点分析
- 竞品分析
- 市场机会

## 3. 产品愿景 (Product Vision)
- 长期愿景
- 成功标准
- 关键指标

## 4. 简化策略 (Simplification Strategy)
- 简化路径选择（价格/命题）
- 选择依据（三项测试结果）
- 简化承诺

## 5. 产品方案 (Product Solution)
- 核心功能清单
- 功能裁剪决策
- 用户体验设计原则
- 技术架构要点

## 6. 执行计划 (Execution Plan)
- 专属系统设计
- 扩张路线图
- 关键里程碑
- 资源需求

## 7. 防御策略 (Defense Strategy)
- 潜在威胁分析
- 防御措施
- 收购/创新计划

## 8. 附录 (Appendix)
- Brainstorming 创意池
- 测试数据
- 参考文档
```

## 各章节撰写指南

### 1. 执行摘要

**目的：** 让读者在 2 分钟内理解产品核心

**内容要求：**
- **一句话产品定义**：清晰描述产品为谁解决什么问题
- **简化路径宣言**：明确选择价格简化还是命题简化
- **核心价值主张**：用数据说明预期影响

**示例：**
```
一句话产品定义：为中小企业提供零门槛的 CRM 工具，让销售管理像发微信一样简单。

简化路径宣言：我们选择命题简化路径，通过极致的用户体验降低使用门槛。

核心价值主张：将 CRM 使用学习成本从 2 周降低到 2 小时，用户激活率提升 300%。
```

### 2. 市场洞察

**目的：** 证明产品定义基于充分的市场理解

**内容要求：**

**目标用户画像：**
```
- 用户类型：[具体描述]
- 核心特征：[3-5 个关键特征]
- 使用场景：[典型使用场景]
- 支付意愿：[价格敏感度]
```

**核心痛点分析：**
```
按优先级列出 Top 3 痛点：
1. [痛点描述] - 影响程度 [高/中/低] - 现有解决方案缺陷
2. ...
3. ...
```

**竞品分析：**
```
| 竞品 | 简化路径 | 优势 | 劣势 | 我们的差异化 |
|------|---------|------|------|-------------|
| 竞品 A | 价格简化 | 低价 | 体验差 | 体验领先 |
| 竞品 B | 命题简化 | 体验好 | 价格高 | 价格更优 |
```

**市场机会：**
```
- 市场规模：[数据支撑]
- 增长趋势：[增长率]
- 空白点：[未被满足的需求]
```

### 3. 产品愿景

**目的：** 描述产品的长期目标和成功标准

**内容要求：**

**长期愿景：**
```
3-5 年后产品是什么样的？
在市场中占据什么位置？
为用户创造什么价值？
```

**成功标准：**
```
短期（6 个月）：
- [可量化指标 1]
- [可量化指标 2]

中期（1-2 年）：
- [市场份额]
- [用户规模]

长期（3-5 年）：
- [行业地位]
- [品牌认知]
```

**关键指标：**
```
北极星指标：[唯一关键指标]

支撑指标：
- 用户获取：[CAC, 获客渠道]
- 用户激活：[激活率，时间]
- 用户留存：[次日/7 日/30 日留存]
- 商业价值：[LTV, ARPU]
```

### 4. 简化策略

**目的：** 详细说明简化路径选择和依据

**内容要求：**

**简化路径选择：**
```
我们选择 [价格简化/命题简化] 路径

核心理由：
1. [倾向测试结果]
2. [市场间隙测试结果]
3. [钥匙测试结果]
```

**倾向测试详情：**
```
团队基因评估：
- 优势领域：[具体能力]
- 资源禀赋：[独特资源]
- 文化特质：[团队文化]
```

**市场间隙测试详情：**
```
竞品扫描结果：
- 价格简化者：[列表及分析]
- 命题简化者：[列表及分析]
- 市场空白：[具体机会点]
```

**钥匙测试详情：**
```
核心优势：
- [独特技术/资源/流程]
- 可持续性分析
- 可复制性评估
```

**简化承诺：**
```
我们对用户的承诺：
[一句话描述简化承诺]

具体体现：
- 成本降低：[具体数据]
- 步骤减少：[具体数据]
- 体验提升：[具体数据]
```

### 5. 产品方案

**目的：** 描述产品的具体方案

**内容要求：**

**核心功能清单：**
```
Must Have（必须有）：
1. [功能名称] - [核心价值] - [优先级 P0]
2. ...

Should Have（应该有）：
1. [功能名称] - [核心价值] - [优先级 P1]
2. ...

Won't Have（本次不做）：
1. [功能名称] - [不做的理由]
2. ...
```

**功能裁剪决策：**
```
裁剪原则：
[基于什么原则决定做/不做]

重大裁剪决策：
- [决定不做的功能] - [决策理由] - [节省成本]
- ...
```

**用户体验设计原则：**
```
核心设计原则：
1. [原则名称] - [具体描述] - [案例]
2. ...

用户旅程关键节点：
[绘制关键节点图或描述]
```

**技术架构要点：**
```
架构选择：
[核心架构决策及理由]

技术选型：
[关键技术选择]

扩展性考虑：
[如何支持未来扩展]
```

### 6. 执行计划

**目的：** 说明如何将战略落地

**内容要求：**

**专属系统设计：**

```
- 供应链设计：[关键措施]
- 成本控制：[具体方法]
- 规模化路径：[阶段目标]

- 用户反馈体系：[收集机制]
- 快速迭代：[迭代周期]
- 体验监控：[指标体系]
```

**扩张路线图：**
```

- 目标：[具体目标]
- 关键动作：[3-5 个]

阶段二（6-12 个月）：
- ...

- ...
```

**关键里程碑：**
```
| 时间 | 里程碑 | 成功标准 | 负责人 |
|------|--------|---------|--------|
| M1 | MVP 发布 | [标准] | [人] |
```

**资源需求：**
```

- [角色] × [人数]

- [阶段] - [金额] - [用途]

- [特殊需求]
```

## 撰写流程

### 步骤 1：收集输入材料

**从 ESP-Thinking 收集：**
- [ ] 创意清单
- [ ] 极简测试结果
- [ ] 多视角意见

**从 ESP-Plan 收集：**
- [ ] 简化路径宣言
- [ ] 功能裁剪清单/用户体验地图
- [ ] 三项测试详情

**从 ESP-Act 收集：**
- [ ] 专属系统设计
- [ ] 扩张路线图
- [ ] 防御策略

### 步骤 2：撰写初稿

**建议顺序：**
1. 先写 2-5 章（核心内容）
2. 再写 6-7 章（执行和防御）
3. 最后写 1 章（执行摘要）
4. 补充 8 章（附录）

**时间分配：**
- 核心内容（2-5 章）：60%
- 执行防御（6-7 章）：25%
- 执行摘要（1 章）：10%
- 附录（8 章）：5%

### 步骤 3：评审与优化

**自检清单：**
- [ ] 文档是否简洁（总长度<30 页）？
- [ ] 简化路径是否清晰？
- [ ] 功能裁剪决策是否有充分理由？
- [ ] 执行计划是否可落地？
- [ ] 防御策略是否具体？
- [ ] 所有数据是否有来源？
- [ ] 语言是否无歧义？

**他检流程：**
1. 业务负责人评审：商业逻辑
2. 技术负责人评审：技术可行性
3. 设计负责人评审：用户体验
4. CEO/决策层评审：战略一致性

### 步骤 4：定稿与发布

**定稿检查：**
- [ ] 所有评审意见已处理
- [ ] 格式统一
- [ ] 图表清晰
- [ ] 链接有效

**发布动作：**
- [ ] 发送给所有相关人员
- [ ] 组织宣讲会议
- [ ] 建立文档版本管理
- [ ] 设定回顾日期（6-12 个月）

## 文档质量标准

### 优秀文档的特征

| 特征 | 表现 | 检查方法 |
|------|------|---------|
| **简洁** | 2 分钟内理解核心 | 给新人阅读，计时测试 |
| **清晰** | 无歧义，决策明确 | 问 3 个人，看理解是否一致 |
| **可执行** | 有具体行动项 | 检查是否有"谁在何时做什么" |
| **可验证** | 有量化指标 | 检查是否有数字和目标 |
| **一致** | 各章节逻辑自洽 | 检查前后是否矛盾 |

### 常见错误

| 错误 | 表现 | 修复 |
|------|------|------|
| **过于冗长** | >50 页，重点不突出 | 强制压缩到 30 页内 |
| **模糊表述** | "可能"、"大概"、"考虑" | 改为确定性表述 |
| **缺乏数据** | 只有定性描述 | 补充量化数据 |
| **决策不明** | 没有明确做/不做 | 明确决策及理由 |
| **执行空洞** | 没有具体行动项 | 补充里程碑和负责人 |

## 使用提示

1. **文档是工具，不是目的**：文档价值在于指导行动，不在于文档本身
2. **保持更新**：市场变化时及时更新文档
3. **版本管理**：使用版本号，记录重大变更
4. **定期回顾**：6-12 个月回顾，验证假设是否成立

通过 ESP-Document，你将把 ESP 三阶段的思考转化为可执行的文档，确保团队对齐、执行有力。
