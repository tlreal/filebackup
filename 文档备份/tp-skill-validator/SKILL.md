# TP-Skill 质量验证器

## Overview

对 TP-Skill 进行结构化质量审查，输出分级审查报告。核心原则：每个检查项必须基于《TP-Skill 撰写规范指南》的明确规则，判定结果必须是客观的（✅通过 / ⚠️建议改进 / ❌必须修复），不允许主观的"感觉还行"。

## When to Use

- 新写的 TP-Skill 准备上传 TPClawHub 之前
- 对铸剑行动中产出的已有 Skill 做质量体检
- TP-Skill-Refiner 完成优化后的最终验证
- 技术经理或项目经理评估团队 Skill 质量时

**When NOT to Use:**
- 需要从零创建一个新 Skill → 使用 Anthropic 的 `skill-creator`
- 需要优化一个已有 Skill 的内容 → 使用 `tp-skill-refiner`
- 只是想了解 Skill 是什么 → 阅读《TP-Skill 撰写规范指南》第一篇

## The Process

### Step 1: 读取待审查的 Skill

1. 读取目标 Skill 文件夹中的 `SKILL.md` 文件
2. 检查文件夹中是否存在 `references/`、`scripts/`、`assets/` 等子目录
3. 如果存在子目录，记录其中的文件列表（后续检查会用到）

**如果目标路径不存在 SKILL.md**：直接终止审查，报告"未找到 SKILL.md 文件，无法进行质量验证"。

### Step 2: 执行结构合规检查（底线层）

对照以下检查规则，逐项验证：

#### 2.1 Frontmatter 检查

| 检查项 | 规则 | 判定标准 |
|-------|------|---------|
| FM-01: Frontmatter 存在 | 文件开头必须有 `---` 包裹的 YAML 块 | 存在=✅ / 不存在=❌ |
| FM-02: name 字段 | 必须存在，只能包含小写英文字母、数字和连字符 | 合规=✅ / 缺失或格式错误=❌ |
| FM-03: name 与文件夹名一致 | name 字段的值必须与 Skill 所在文件夹名相同 | 一致=✅ / 不一致=❌ |
| FM-04: description 字段 | 必须存在，长度不少于 20 字 | 合规=✅ / 缺失或过短=❌ |
| FM-05: description 无流程泄漏 | description 中不应包含"先…再…然后…"等流程描述词汇 | 无泄漏=✅ / 有泄漏=❌ |
| FM-06: description 触发场景 | 应以触发场景描述开头（如"用于在…时使用"或"Use when"） | 合规=✅ / 不符=⚠️ |

#### 2.2 正文结构检查

| 检查项 | 规则 | 判定标准 |
|-------|------|---------|
| ST-01: Overview 章节 | 必须存在，内容 1-3 句话 | 存在=✅ / 缺失=❌ |
| ST-02: When to Use 章节 | 必须存在，至少列举 2 个适用场景 | 存在且≥2项=✅ / 缺失=❌ / 仅1项=⚠️ |
| ST-03: When NOT to Use | 建议存在，说明不适用场景 | 存在=✅ / 缺失=⚠️ |
| ST-04: The Process 章节 | 必须存在，至少包含 2 个步骤 | 存在且≥2步=✅ / 缺失=❌ / 仅1步=⚠️ |
| ST-05: 质量检查点 | 强烈建议存在，至少 3 条 | 存在且≥3条=✅ / 存在但<3条=⚠️ / 缺失=⚠️ |
| ST-06: Common Mistakes | 强烈建议存在 | 存在=✅ / 缺失=⚠️ |
| ST-07: 产出物规范 | 建议存在 | 存在=✅ / 缺失=⚠️ |

### Step 3: 执行内容质量检查（底线层 + 目标层）

#### 3.1 流程可执行性检查

逐步审查 The Process 中的每个步骤：

| 检查项 | 规则 | 判定标准 |
|-------|------|---------|
| EX-01: 步骤清晰度 | 每步必须明确"做什么"，不能是"处理数据"、"分析问题"等模糊描述 | 全部清晰=✅ / 有模糊步骤=❌ |
| EX-02: 判据量化 | 涉及判断的步骤应提供具体量化标准（如"≤ -130 dBm"），而非"正常范围" | 全部量化=✅ / 有模糊判据=⚠️ |
| EX-03: 异常处理 | 至少有 1 个步骤包含异常情况的处理说明 | 存在=✅ / 完全没有=❌ |
| EX-04: 祈使句风格 | 步骤描述应使用祈使句（"检查…"、"读取…"），而非"您可以考虑…" | 合规=✅ / 有不当用语=⚠️ |

#### 3.2 防幻觉检查

| 检查项 | 规则 | 判定标准 |
|-------|------|---------|
| AH-01: 专有数据提供 | 涉及公司产品参数、错误码等专有数据时，必须在 Skill 中显式提供或指向 references/ 中的文件 | 已提供=✅ / 依赖AI背景知识=❌ |
| AH-02: 红线提示 | 建议包含"不要编造"类型的约束提示 | 存在=✅ / 缺失=⚠️ |
| AH-03: 数据来源标注 | 建议要求 AI 在产出中标注数据来源 | 存在=✅ / 缺失=⚠️ |

#### 3.3 技象规范一致性

| 检查项 | 规则 | 判定标准 |
|-------|------|---------|
| TP-01: 产品术语准确 | 使用正确的产品型号：TP6300（象芯三号）、TP9102（基站）、TPaaS（云平台）等 | 准确=✅ / 有错误=❌ |
| TP-02: 术语统一性 | 同一概念在全文中使用统一的称呼（不混用"基站"/"BS"/"TP9102"） | 统一=✅ / 不统一=⚠️ |
| TP-03: 首次术语标注 | 专业术语首次出现时标注英文全称或中文解释 | 已标注=✅ / 未标注=⚠️ |

### Step 4: 执行 Token 效率检查（目标层）

| 检查项 | 规则 | 判定标准 |
|-------|------|---------|
| TK-01: SKILL.md 行数 | 简单型≤100行，标准型≤300行，复杂型≤500行 | 在范围内=✅ / 超出=⚠️ |
| TK-02: references 分离 | 超过 30 行的数据表应放在 references/ 中，而非内联在正文 | 已分离=✅ / 应分离未分离=⚠️ |

### Step 5: 汇总审查结果并生成报告

生成结构化审查报告，格式如下：

```markdown
# TP-Skill 质量审查报告

**审查对象：** [Skill 名称]
**审查日期：** [YYYY-MM-DD]
**SKILL.md 路径：** [文件路径]
**SKILL.md 行数：** [N 行]

## 审查结果汇总

| 状态 | 数量 | 占比 |
|------|------|------|
| ✅ 通过 | X | XX% |
| ⚠️ 建议改进 | X | XX% |
| ❌ 必须修复 | X | XX% |

| ... | ... | ... | ... |

### 流程可执行性检查

  - 合格：底线层全部通过
```

### Step 6: 提供修复指引

对于每个 ❌ 必须修复的问题，给出具体的修复建议：

1. **指出问题在 SKILL.md 中的具体位置**（行号或章节）
2. **给出修复的方向或示例**（如"description 应改为以'用于在…时使用'开头"）
3. **如果问题适合自动修复**，建议使用 `tp-skill-refiner` 进行优化

⚠️ 红线提示：
- 不要降低底线标准来让 Skill "通过"审查
- 底线层的 ❌ 项不可以降级为 ⚠️
- 如果不确定某项是否通过，标记为 ⚠️ 并说明不确定的原因

## 产出物规范

| 产出 | 格式 | 存放位置 |
|------|------|---------|
| 质量审查报告 | Markdown | 与被审查 Skill 同级目录下，命名为 `review-YYYY-MM-DD.md` |

## 质量检查点

- [ ] 所有检查项（FM/ST/EX/AH/TP/TK）均已逐项评估
- [ ] 每个 ❌ 项都有具体的问题描述和修复建议
- [ ] 审查报告包含总体评价和是否可上传 TPClawHub 的明确判定
- [ ] 没有遗漏 SKILL.md 中的任何章节
- [ ] 改进建议清单是可操作的（不是泛泛而谈）

## Common Mistakes

| 错误做法 | 正确做法 |
|---------|---------|
| 只看 Frontmatter 不看正文内容 | 必须逐步审查 The Process 中每个步骤的质量 |
| 把所有问题都标为 ⚠️ 来"放水" | 底线层问题必须严格按规则标 ❌ |
| 审查报告只写"通过/不通过" | 每条都要附具体说明和修复建议 |
| 忽略 references/ 中的文件 | 如果正文引用了 references/，要检查引用的文件是否实际存在 |
| 用主观感受代替规则判定 | 每项判定必须引用具体的检查规则（FM-01、ST-02 等） |
