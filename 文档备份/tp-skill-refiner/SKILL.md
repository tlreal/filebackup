# TP-Skill 优化器

## Overview

将初版或不规范的 TP-Skill 按照技象标准进行结构化优化。核心原则：**保留原始 Skill 的核心意图和业务逻辑，只优化结构和质量**——不替换作者的专业判断，只帮他把想法表达得更好。

## When to Use

- 铸剑行动中产出的初版 Skill 需要升级为合格 Skill
- 从网上或其他来源获取的 Skill 需要适配技象规范
- tp-skill-validator 审查报告中有 ❌ 或多项 ⚠️ 需要修复时
- 把已有的 Prompt/操作手册转化为标准的 TP-Skill

**When NOT to Use:**
- 需要从零创建一个新 Skill → 使用 Anthropic 的 `skill-creator`
- 只需要检查 Skill 质量，不需要修改 → 使用 `tp-skill-validator`
- Skill 的业务逻辑本身有问题（如错误的参数值）→ 需要领域专家确认后再优化

## The Process

### Step 1: 理解原始 Skill 的意图

1. 完整阅读待优化的 SKILL.md（以及 references/ 中的文件，如果有的话）
2. 回答以下问题（内部思考，不需要输出）：
   - 这个 Skill 要解决什么问题？
   - 它的目标用户是谁？（研发？工程？非技术？）
   - 它的核心业务逻辑是什么？
   - 它目前最大的质量问题是什么？

**红线**：如果无法理解原始 Skill 的意图，停下来询问用户"这个 Skill 的主要目的是什么？"。不要凭猜测进行优化。

### Step 2: 运行 tp-skill-validator 获取基线报告

在优化之前，先用 `tp-skill-validator` 的检查规则对原始 Skill 做一次评估（不需要输出完整报告，内部记录各项检查结果即可）。

记录：
- 有多少项 ❌（必须修复）
- 有多少项 ⚠️（建议改进）
- 当前质量等级是什么

这为后续优化提供了明确的改进目标。

### Step 3: 修复 Frontmatter

按照以下规则修复或创建 Frontmatter：

1. **name 字段**
   - 如果没有：根据 Skill 内容生成，格式为小写英文+连字符
   - 如果格式不对：转换为正确格式（去掉中文、大写、下划线）
   - 确保与文件夹名一致

2. **description 字段**
   - 如果没有：根据 Skill 内容撰写
   - 如果有流程泄漏：重写为纯触发场景描述
   - 格式要求：以"用于在…时使用"或类似触发场景描述开头
   - 列举 2-3 个具体的触发场景
   - 不包含任何工作流程描述

### Step 4: 补全正文结构

按照以下标准章节结构，补全缺失的部分：

```markdown
# [技能名称]

## Overview
## When to Use
## When NOT to Use
## The Process
### Step 1: ...
### Step 2: ...
## 产出物规范
## 质量检查点
## Common Mistakes
```

**补全规则**：

| 缺失章节 | 如何补全 |
|---------|---------|
| Overview | 从原始 Skill 内容中提炼 1-2 句核心描述 |
| When to Use | 从 description 和正文中提取适用场景 |
| When NOT to Use | 根据 Skill 的边界推断不适用场景 |
| The Process | **保留原始流程逻辑**，重新组织为编号步骤 |
| 产出物规范 | 从流程结尾推断最终产出物 |
| 质量检查点 | 根据流程的关键产出设计 3-5 条检查项 |
| Common Mistakes | 根据流程中的判断节点和异常路径推断常见错误 |

⚠️ **保留原则**：
- **保留**原始 Skill 中所有的业务逻辑、判据数值、专有数据
- **保留**原始 Skill 中已有的好的内容和表述
- **不替换**与领域专业相关的判断（如"频偏阈值 ±2 ppm"——这是作者的专业判断，不要改）
- **只重组**结构，**只补全**缺失部分

### Step 5: 优化流程步骤质量

对 The Process 中的每个步骤进行质量提升：

1. **模糊步骤 → 具体步骤**
   - 原始："分析数据" → 优化："读取 CSV 文件中的频偏列，逐行检查是否在 ±2 ppm 范围内"
   - 注意：只有在不改变原始意图的前提下才具体化

2. **补充异常处理**
   - 对每个判断节点，添加"如果…怎么办"
   - 默认策略："停下来告知用户，请求确认"

3. **补充祈使句风格**
   - "您可以考虑读取…" → "读取…"
   - "建议您先检查…" → "检查…"

4. **补充防幻觉措施**
   - 如果步骤中引用了公司产品参数但没有显式提供数值 → 添加红线提示
   - 添加"不要编造未在本文档中提供的数据"的约束

### Step 6: 优化术语一致性

1. 扫描全文，列出所有技象产品相关术语
2. 统一为标准称呼：
   - TP6300 / 象芯三号 → 首次出现写"TP6300（象芯三号）"，后续用"TP6300"
   - TP9102 / 基站 → 首次"TP9102 基站"，后续用"TP9102"
   - TPaaS / 云平台 → 首次"TPaaS 云平台"，后续用"TPaaS"
3. 专业术语首次出现时标注中英文

### Step 7: 输出优化后的 Skill

1. 输出完整的优化后 SKILL.md
2. 在文件末尾添加变更记录：
   ```markdown
   ## 变更历史

   | 版本 | 日期 | 修改人 | 修改内容 |
   |------|------|-------|---------|
   | 1.0 | [原始日期] | [原作者] | 初版 |
   | 1.1 | [今天日期] | TP-Skill-Refiner | 按照技象规范优化结构和质量 |
   ```

### Step 8: 运行 tp-skill-validator 验证

对优化后的 Skill，再次按照 `tp-skill-validator` 的检查规则评估。

**验证目标**：
- 底线层所有 ❌ 项变为 ✅
- 质量等级至少达到"🥈 良好"
- 对比 Step 2 的基线报告，确认有明显改进

**如果仍有 ❌ 项**：返回 Step 4 或 Step 5 继续修复，直到底线层全部通过。

向用户报告优化前后的质量对比：

```markdown
## 优化效果对比

| 指标 | 优化前 | 优化后 |
|------|-------|-------|
| ✅ 通过 | X 项 | X 项 |
| ⚠️ 建议改进 | X 项 | X 项 |
| ❌ 必须修复 | X 项 | 0 项 |
| 质量等级 | 🥉/🥈/🥇 | 🥉/🥈/🥇 |
| SKILL.md 行数 | X 行 | X 行 |
```

## 产出物规范

| 产出 | 格式 | 说明 |
|------|------|------|
| 优化后的 SKILL.md | Markdown | 覆盖原始文件或保存为新版本 |
| 优化效果对比 | Markdown 表格 | 展示优化前后的质量变化 |

## 质量检查点

- [ ] 原始 Skill 的所有业务逻辑和专有数据都被保留
- [ ] 优化后的 Skill 通过 tp-skill-validator 底线层全部检查
- [ ] Frontmatter 格式正确，description 无流程泄漏
- [ ] 至少包含 Overview、When to Use、The Process、质量检查点 四个章节
- [ ] 术语在全文中使用统一
- [ ] 至少有 1 个步骤包含异常处理说明
- [ ] 没有擅自修改作者的专业判断（参数值、判据标准等）

## Common Mistakes

| 错误做法 | 正确做法 |
|---------|---------|
| 重写整个 Skill 的业务逻辑 | 只重组结构，保留原始业务逻辑 |
| 删除原始 Skill 中"语句不通顺"的专业表述 | 保留专业内容，只修正结构问题 |
| 编造 Skill 中没有的判据值 | 只使用原始 Skill 中已有的数值，或标注"待作者补充" |
| 优化后不做验证 | 必须用 tp-skill-validator 的规则再验证一次 |
| 把所有 Skill 优化成同一个模板 | 根据类型（研发/工程/非技术）采用不同的侧重点 |
