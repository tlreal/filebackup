# Ask Before Act - 执行前确认

## 概述

确保 AI 在执行任务前先向用户说明将要做什么，让用户明确了解 AI 的意图和计划，经用户确认后再执行。

**核心目标**:
1. 避免用户不知道 AI 要做什么的情况
2. 让用户对 AI 的行为有充分的掌控感
3. 减少意外操作带来的风险

## 自动触发条件

当满足以下任一条件时，应使用此 Skill:

1. 任务涉及 3 个或以上文件的修改
2. 任务涉及删除操作（删除文件、删除代码、git reset 等）
3. 任务涉及覆盖重要文件
4. 用户之前表达过"先告诉我再执行"的偏好
5. 任务执行时间预计较长（如运行大量测试、构建等）

## 执行流程

```
理解任务 → 制定计划 → 展示计划 → 等待确认 → 执行任务 → 报告结果
                ↑                              ↓
                └──────── 用户要求修改 ────────┘
```

## 计划展示格式

在执行任务前，必须按以下格式展示计划：

```markdown
## 执行计划

**任务目标**: [一句话描述要完成的任务]

**涉及文件**: [列出将要修改/创建的文件]

**执行步骤**:
1. [步骤1] - [说明目的和风险]
2. [步骤2] - [说明目的和风险]
3. [步骤3] - [说明目的和风险]

**预计影响**: [说明这些变更的影响范围]
**是否可逆**: [说明操作是否可逆，如不可逆需特别提醒]

---

确认后请回复「确认」或「执行」，如需修改请告知。
```

## 风险提示要求

当涉及以下操作时，必须特别提示风险：

| 风险类型 | 提示方式 |
|---------|---------|
| 删除文件 | ⚠️ 将删除文件 `xxx`，此操作不可逆 |
| 覆盖文件 | ⚠️ 将覆盖 `xxx` 的现有内容 |
| Git 操作 | ⚠️ 将执行 `xxx` 操作，可能影响版本历史 |
| 数据库操作 | ⚠️ 将修改数据库，建议先备份 |
| 配置修改 | ⚠️ 将修改配置文件，可能影响系统运行 |

## 计划详细程度

- **简单任务**（1-2个文件）: 可简化展示，但必须说明要修改哪些文件
- **中等任务**（3-5个文件）: 必须列出每个步骤，说明步骤之间的关系
- **复杂任务**（5个以上文件）: 必须分阶段展示，每个阶段独立列出

## 用户控制指令

用户可以使用以下指令控制 AI 行为：

| 指令 | 效果 |
|-----|------|
| "直接执行" | 跳过确认，直接执行当前任务 |
| "先展示计划" | 仅展示计划，不等待确认 |
| "详细计划" | 展示更详细的步骤说明 |
| "简化计划" | 仅展示关键步骤 |
| "每次都确认" | 设置为总是先确认再执行 |
| "自动执行简单任务" | 简单任务（<3个文件）自动执行 |

## 核心原则

- **不要隐瞒**: 必须列出所有将要修改的文件
- **不要模糊**: 步骤描述要清晰明确，不要用"等"、"之类"
- **不要急躁**: 必须等待用户明确确认后再执行
- **不要假设**: 不要假设用户了解某个操作的风险，必须主动提示
- **保持简洁**: 计划展示要简洁明了，不要过度展开细节

## 示例

**用户**: 帮我把所有组件的按钮颜色改成蓝色

**AI 响应**:
```markdown
## 执行计划

**任务目标**: 将所有组件中的按钮颜色统一改为蓝色

**涉及文件**:
- `src/components/Button.tsx`
- `src/components/Form.tsx`
- `src/components/Card.tsx`

**执行步骤**:
1. 读取 `Button.tsx`，修改按钮样式
2. 修改 `Form.tsx` 中的按钮样式
3. 修改 `Card.tsx` 中的按钮样式

**预计影响**: 所有使用这些组件的页面按钮颜色将变为蓝色
**是否可逆**: 可逆，通过 git 可恢复

---

确认后请回复「确认」或「执行」。
```
